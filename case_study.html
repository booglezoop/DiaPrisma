Case Study · Solo-built, self-validated

A qualification layer for sellers who aren't ready - before an agency wastes time finding out.

DiaPrisma scores and routes real estate leads for Bulgarian agencies. Sole designer, builder, and engineer: Iliyan Donev.

Vanilla JS Netlify Functions Supabase / Postgres n8n Gmail API

The problem

Agencies treat every inbound seller inquiry the same way - regardless of whether that seller is actually ready to sell, or just curious.

DiaPrisma sits in front of an agency's own traffic as a qualification and routing layer, not a lead-generation tool. A seller answers a short branching quiz. They get an immediate readiness score and a diagnosed risk. If they opt in, the agency receives a structured, pre-scored lead instead of a raw form submission - tiered by urgency, not treated as one undifferentiated inbox.

Architecture

Client-side quiz → server-side re-validation → database → three decoupled automation workflows.

03

↓ webhook
04
n8n - real-time routing
tier × track switch → 4 Gmail templates
05
n8n - daily digest
cron 09:00, batches low-priority leads into one email
06
n8n - reconciliation watchdog
cron /4h, catches silently-failed real-time notifications

The real-time path and the watchdog are deliberately decoupled, so one failed webhook doesn't mean a hot lead disappears silently - it gets caught and re-alerted within a bounded window.

Notable design decisions

Two independent scores, not one

A seller-facing readiness score and a separate agent-facing lead-priority score, computed from the same raw answers but structurally decorrelated - neither feeds into the other. A seller can be poorly prepared but highly motivated, or well-prepared and in no hurry. Agency action priority should track motivation, not readiness, and keeping the two independent avoids a shortcut that would quietly conflate them.

The server never trusts the client

The client does UX-layer validation for responsiveness. The Netlify function re-validates every field independently, re-checks score ranges, and rate-limits by IP against a live Supabase read before insert - so a tampered payload can't inject an out-of-range score or skip the client entirely.

Defense in depth, scoped honestly

Input sanitization closes the main stored-XSS path into downstream email templates. It's also documented that if a future field were added without running through sanitization, there's no second layer of escaping at the template level - flagged as an open item rather than assumed away.

GDPR retention, enforced - not just documented

A pg_cron job scrubs each row's IP address 30 days after submission and expires the row entirely at 12 months, matching what the privacy policy actually states. This is a data-minimization control, not a demo/test-data filter - it doesn't distinguish a real seller's row from a test submission, and isn't meant to; it applies the same retention rule to every row regardless of source.

What I'd flag as an engineer

Auditing this system myself - rather than presenting it as finished - turned up real issues. Here's what's still open, and one that's already fixed.

DIA-01

Secondary risk-messaging logic bug

A missing block scope meant a mispricing-secondary narrative silently gated an unrelated documentation check behind an extra condition - certain sellers with undocumented paperwork never saw that flagged. Root cause documented before patching.

Fixed
DIA-02

No tier/score cross-validation at the API boundary

The server checks that scores fall in range, but not that the submitted lead tier is internally consistent with the score. A malformed or tampered payload could in theory claim a tier the score doesn't support. Low practical risk at current scale - real gap.

Open
DIA-03

Reconciliation watchdog interval is untested

The watchdog pattern is in place and structurally sound - a failed real-time webhook gets caught within a bounded window. That window has a ~4-hour worst case, and hasn't been stress-tested against an actual webhook failure yet.

Open
The bigger open question isn't in this list. This system was designed, built, and audited end-to-end by one person, without a live agency partner supplying real seller traffic. That means the scoring weights and tier thresholds are validated for internal logic - do the rules do what I intended - but not for whether they produce sensible outcomes against real seller behavior. I'm treating that as a known, permanent limitation of building this independently, not as a phase that closes once a pilot starts. The bugs above are fixable in minutes; not knowing whether the model reflects reality is the honest constraint of engineering a system outside of a live business.
